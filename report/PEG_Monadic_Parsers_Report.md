# Формальная часть проекта: PEG как монадические парсеры на Haskell

## 1. Постановка задачи

**Вход:** файл с PEG-грамматикой `G` и строка `w`.

**Выход:** ответ, принадлежит ли строка `w` языку `L(G)`.

Строка считается принятой, если стартовое правило успешно разбирает весь вход, то есть после разбора позиция находится в конце строки.

Дополнительно проект умеет генерировать Haskell-модуль, где каждое правило PEG превращается в монадический парсер типа `Parser ()`.

## 2. Формат входной грамматики

В проекте используется подмножество синтаксиса Peggy.js: без семантических действий и JavaScript-предикатов.

```ebnf
Grammar     ::= Rule+
Rule        ::= Identifier "=" Expression ";"?
Expression  ::= Sequence ("/" Sequence)*
Sequence    ::= Prefix*
Prefix      ::= "&" Prefix | "!" Prefix | Suffix
Suffix      ::= Primary ("*" | "+" | "?")*
Primary     ::= Identifier
              | StringLiteral
              | CharClass
              | "."
              | "(" Expression ")"
```

Поддерживаются строковые литералы в одинарных и двойных кавычках, символьные классы вида `[a-z]` и `[^0-9]`, а также точка `.` для любого одного символа.

Пустое выражение можно записать как `""`, `eps`, `epsilon` или `ε`.

Если в CLI не указан параметр `--start`, стартовым считается первое правило в файле.

## 3. Абстрактный синтаксис PEG

После разбора грамматика переводится в AST:

```haskell
data Expr
  = Epsilon
  | Literal String
  | AnyChar
  | CharClass Bool [ClassAtom]
  | NonTerminal Name
  | Sequence [Expr]
  | Choice [Expr]
  | And Expr
  | Not Expr
  | Many Expr
  | Some Expr
  | Optional Expr
```

Важно, что `Choice` — это **упорядоченный** выбор. Это не обычная симметричная альтернатива: если первая ветка успешно разобрала префикс, остальные ветки уже не пробуются.

## 4. Денотационная семантика

Пусть `w` — входная строка длины `n`, а `i` — текущая позиция, где `0 <= i <= n`.

Для выражения `e` определим частичную функцию:

```text
[[e]]_G(w, i) ∈ {Fail} ∪ {Ok j | i <= j <= n}
```

`Ok j` означает, что выражение `e` успешно разобрало часть строки `w[i..j)` и оставило позицию `j`.

`Fail` означает отказ.

Базовые случаи:

```text
[[ε]](w, i) = Ok i

[["s"]](w, i) = Ok (i + |s|), если s является префиксом w[i..]
              = Fail, иначе

[[.]](w, i) = Ok (i + 1), если i < |w|
            = Fail, иначе

[[A]]_G(w, i) = [[R(A)]]_G(w, i),
где R(A) — правая часть правила A
```

Последовательность:

```text
[[e1 e2]](w, i) =
  если [[e1]](w, i) = Ok j, то [[e2]](w, j),
  иначе Fail.
```

Упорядоченный выбор:

```text
[[e1 / e2]](w, i) =
  [[e1]](w, i), если [[e1]](w, i) = Ok j;
  [[e2]](w, i), если [[e1]](w, i) = Fail.
```

Просмотр вперёд:

```text
[[&e]](w, i) = Ok i, если [[e]](w, i) успешно;
             = Fail, иначе.

[[!e]](w, i) = Ok i, если [[e]](w, i) = Fail;
             = Fail, иначе.
```

Повторы:

```text
[[e*]](w, i) = результат жадного повторения e,
               пока e успешно потребляет хотя бы один символ.

[[e+]](w, i) = [[e e*]](w, i).

[[e?]](w, i) = [[e / ε]](w, i).
```

Если выражение `e` может успешно завершиться без продвижения по входу, то `e*` и `e+` запрещаются валидатором. Иначе жадный повтор может зациклиться.

Язык грамматики задаётся так:

```text
L(G) = { w | [[S]]_G(w, 0) = Ok |w| },
```

где `S` — стартовое правило.

## 5. Монадический парсер

В коде парсер представлен типом:

```haskell
newtype Parser a = Parser
  { runParser :: RunState -> Either String (Maybe a, RunState) }
```

`RunState` хранит исходную строку, текущую позицию и memo-таблицу.

`Maybe a` отвечает за обычный успех или отказ парсера, а `Either String` — за аварийные ошибки, например за обнаруженный во время выполнения zero-width-цикл.

Для `Parser` реализованы стандартные классы:

```haskell
Functor Parser
Applicative Parser
Monad Parser
Alternative Parser
```

Идея такая:

- `pure x` — успешный парсер без потребления входа;
- `>>=` — последовательный запуск парсеров;
- `empty` — отказ;
- `<|>` — PEG-выбор: сначала левая ветка, правая пробуется только при отказе левой, с откатом позиции.

Основные комбинаторы:

```haskell
literal   :: String -> Parser ()
anyChar   :: Parser Char
charClass :: Bool -> [ClassAtom] -> Parser ()
look      :: Parser a -> Parser ()
neg       :: Parser a -> Parser ()
manyP     :: Parser () -> Parser ()
someP     :: Parser () -> Parser ()
optionalP :: Parser () -> Parser ()
named     :: String -> Parser () -> Parser ()
```

## 6. Компиляция AST в парсер

AST компилируется в парсер структурной рекурсией по выражению:

```text
C[ε]          = pure ()
C["s"]        = literal s
C[.]          = anyChar *> pure ()
C[[...]]      = charClass ...
C[A]          = named "A" C[R(A)]
C[e1 ... ek]  = C[e1] *> ... *> C[ek]
C[e1 / ...]   = choose [C[e1], ...]
C[&e]         = look C[e]
C[!e]         = neg C[e]
C[e*]         = manyP C[e]
C[e+]         = someP C[e]
C[e?]         = optionalP C[e]
```

Проверка принадлежности строки запускает `C[S] *> endOfInput` из позиции `0`.

То есть строка принимается только тогда, когда стартовое правило успешно разобрало весь вход.

## 7. Packrat-мемоизация

Для каждого вызова нетерминала `A` в позиции `i` сохраняется результат:

```text
memo[(A, i)] = InProgress | Done Fail | Done (Ok j)
```

Алгоритм `named A p` работает так:

1. Если `(A, i)` уже вычислено, возвращается сохранённый результат.
2. Если `(A, i)` сейчас находится в состоянии `InProgress`, значит найден цикл или левая рекурсия.
3. Иначе пара `(A, i)` помечается как `InProgress`, выполняется тело правила, а затем результат сохраняется в memo-таблице.

За счёт этого рекурсивные вызовы в одной и той же позиции не пересчитываются заново. Это и есть основная идея packrat-парсинга для PEG.

## 8. Валидация

Перед запуском грамматика проходит несколько проверок:

1. Нет дублирующихся правил.
2. Все используемые нетерминалы определены.
3. Стартовый нетерминал существует.
4. Нет левой рекурсии.
5. В `e*` и `e+` выражение `e` не может успешно завершиться без потребления входа.

Для последних двух пунктов строится консервативная аппроксимация `zeroProgress(A)`: может ли правило `A` успешно завершиться, не сдвинув позицию.

Затем строится граф первых zero-width-вызовов нетерминалов. Если нетерминал достижим из самого себя в этом графе, грамматика считается лево-рекурсивной или потенциально зацикливающейся.

## 9. Корректность: набросок доказательства

**Лемма 1.**  
Если скомпилированный парсер `C[e]`, запущенный на строке `w` из позиции `i`, успешно завершается в позиции `j`, то:

```text
[[e]]_G(w, i) = Ok j
```

Доказательство проводится структурной индукцией по `e`.

Для литералов, точки и символьных классов утверждение следует из определения примитивных парсеров. Для последовательности используется индукционное предположение для каждого элемента. Для выбора важно, что `<|>` пробует правую ветку только после отказа левой и откатывает позицию. Для `&` и `!` используется то, что `look` и `neg` всегда восстанавливают исходную позицию.

**Лемма 2.**  
Если денотационная семантика даёт:

```text
[[e]]_G(w, i) = Ok j
```

то парсер `C[e]`, запущенный из позиции `i`, тоже успешно завершается в позиции `j`.

Доказательство аналогично идёт по структуре выражения. Для нетерминалов используется детерминированность PEG: результат правила в фиксированной позиции единственный, поэтому memo-таблица не меняет поведение, а только переиспользует уже найденный результат.

**Теорема.**  
CLI печатает `ACCEPTED` тогда и только тогда, когда:

```text
w ∈ L(G)
```

Это следует из двух лемм для стартового выражения `S` и дополнительной проверки `endOfInput`. Успех возможен ровно в случае:

```text
[[S]]_G(w, 0) = Ok |w|
```

## 10. Сложность

Пусть `N` — число правил, а `n` — длина входной строки.

Если грамматика не содержит запрещённых zero-width-повторов и левой рекурсии, каждая пара `(нетерминал, позиция)` вычисляется не более одного раза.

Поэтому число вычислений правил ограничено:

```text
O(N * n)
```

Память memo-таблицы также занимает:

```text
O(N * n)
```

Стоимость проверки терминалов зависит от реализации строк. В теоретической оценке предполагается, что доступ к символу и проверка префикса стоят пропорционально длине проверяемого литерала.

В текущей реализации используется `String`, потому что этого достаточно для учебного проекта. Для больших входов его можно заменить на `Text` или массив символов.

## 11. Ограничения

В проекте сознательно не реализованы некоторые возможности полноценного Peggy.js:

- семантические действия;
- метки;
- JavaScript-предикаты;
- поддержка левой рекурсии.

Лево-рекурсивные PEG-грамматики отклоняются валидатором.

Проверка zero-width-ситуаций сделана консервативно: некоторые безопасные грамматики с предикатами могут быть отвергнуты. Зато интерпретатор не зациклится на жадных повторах.

Генератор Haskell-кода также предполагает, что имена правил после санитарной замены не конфликтуют.

## 12. Воспроизведение сборки и проверки

Основная инструкция для запуска находится в `README.md`. Минимальная проверка:

```bash
cabal build
cabal test all
```

Полная проверка одной командой:

```bash
make check
```

Она собирает проект, запускает test-suite, проверяет CLI на примерах из
`examples/`, печатает нормализованную грамматику через `--ast`, генерирует
Haskell-модуль через `--emit-hs` и проверяет этот модуль командой
`ghc -fno-code`.

Ожидаемый результат test-suite:

```text
All tests passed
1 of 1 test suites (1 of 1 test cases) passed.
```
